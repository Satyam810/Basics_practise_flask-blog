# 📝 Flask Blog Platform — Basics Practice Project

A **beginner-friendly Blog Application** built while learning **Flask** from scratch.  
This is a hands-on practice project focused on understanding core Flask concepts — routing, templating, ORM, and authentication — with a **simple, minimal frontend** using plain HTML and Jinja2 (no CSS frameworks).

> 💡 Built as part of backend fundamentals practice during B.Tech (CSE - AI & ML).  
> The goal was **not** a polished UI — it was to deeply understand how Flask, SQLAlchemy, and session-based auth work together under the hood.

---

## 🚀 Features

- 🔐 User Registration
- 🔑 User Login & Logout
- ✍️ Create Blog Posts
- 📝 Edit Your Own Blog Posts
- ❌ Delete Your Own Blog Posts
- 🗂 View All Blog Posts
- 🔒 Secure Password Hashing
- 📦 SQLite Database via SQLAlchemy ORM
- 🧩 Jinja2 Template Inheritance
- 🖥 Simple Plain HTML Frontend (no Bootstrap, no Tailwind — intentional)

---

## 🛠 Tech Stack

| Layer | Technology | Purpose |
|---|---|---|
| Language | Python 3.x | Core language |
| Web Framework | Flask | Routing, views, sessions |
| ORM | Flask-SQLAlchemy | Database models & queries |
| Database | SQLite | Lightweight local DB (via `instance/`) |
| Security | Werkzeug | `generate_password_hash` / `check_password_hash` |
| Templating | Jinja2 | Server-side HTML rendering, template inheritance |
| Frontend | Plain HTML5 | Minimal, no CSS framework — focus on backend |
| Auth | Flask Session | Cookie-based login state management |
| Dev Tools | VS Code, Git, venv | Development environment |

> 🎯 **Why no CSS framework?**  
> This project is intentionally kept to basics. Adding Bootstrap or Tailwind would shift focus away from learning Flask internals — which was the whole point.

```

```

## 🔐 Authentication System

Passwords are hashed using **Werkzeug's `generate_password_hash`** before storing in the DB — plain text passwords are never saved. On login, `check_password_hash` compares the input against the stored hash securely.

Login state is managed via **Flask sessions** (server-side cookie). Protected routes check for an active session before allowing access, and users can only edit or delete **their own** blog posts — authorization is enforced at the route level.

## 🗺 Routes Overview

| Method | Route | Auth Required | Description |
|---|---|---|---|
| GET | `/` | ❌ | Homepage — all blog posts |
| GET/POST | `/register` | ❌ | User registration |
| GET/POST | `/login` | ❌ | User login |
| GET | `/logout` | ✅ | Logout & clear session |
| GET/POST | `/create` | ✅ | Create a new post |
| GET/POST | `/update/<id>` | ✅ Owner only | Edit an existing post |
| GET/POST | `/delete/<id>` | ✅ Owner only | Delete a post |
| GET | `/view/<id>` | ❌ | View a single post |
| GET | `/database` | ✅ | Debug — view all DB records |

## 🎯 What I Practiced & Learned

This project was a deliberate basics drill. Key concepts covered:

- **Flask fundamentals** — app creation, `@app.route`, `request`, `redirect`, `url_for`, `render_template`
- **SQLAlchemy ORM** — defining models, relationships, `db.session`, querying with `.filter_by()`
- **Session management** — storing user state in `session`, protecting routes, logout flow
- **Password security** — hashing before DB insert, verifying on login, never storing raw passwords
- **Jinja2 templating** — `{% extends %}`, `{% block %}`, `{{ variable }}`, `{% if %}` / `{% for %}`
- **Authorization vs Authentication** — knowing who you are vs. what you're allowed to do
- **CRUD with a real DB** — create, read, update, delete tied to actual SQLite records
- **Project organization** — splitting routes into separate files (`blog.py`), keeping `app.py` clean

## 🔄 Next Version — Rebuilt in Django

After completing this Flask basics version, the same project was rebuilt in **Django** to compare how a batteries-included framework handles the same problems:

- Django ORM with migrations (`makemigrations` / `migrate`)
- Built-in Django Authentication (`django.contrib.auth`)
- Admin panel at `/admin` — out of the box
- Blog ownership validation
- Comment system
- MTV (Model-Template-View) structure
- Superuser management via `createsuperuser`

> Flask taught me *how* auth and ORM work from first principles. Django showed me *how frameworks abstract* those same concepts at scale.
